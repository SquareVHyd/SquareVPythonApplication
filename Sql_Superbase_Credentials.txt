That means Supabase is giving you an IPv6-only endpoint, but your network or OS cannot actually reach IPv6 hosts (common on many ISPs and corporate networks).
That’s why ping and pgAdmin fail — they can’t connect over IPv6.

Why This Happens
Supabase’s db.<project>.supabase.co resolves to IPv6 first.
Your system tries IPv6 but fails because:
Your ISP doesn’t support IPv6.
Your router doesn’t have IPv6 enabled.
Your firewall blocks IPv6 traffic.
Solutions
Option 1 — Force IPv4 Resolution
Supabase also has IPv4 addresses, but your DNS resolver is not returning them.
You can try:

On Windows:

Edit C:\Windows\System32\drivers\etc\hosts
Add a line:

Copy code
<IPv4_ADDRESS> db.cztgpruvcuedkoqzniie.supabase.co
To get <IPv4_ADDRESS>, use an online DNS lookup tool that supports IPv4 (e.g., dig from a Linux server with IPv4):
Sh

Copy code
dig A db.cztgpruvcuedkoqzniie.supabase.co @8.8.8.8
Save and retry in pgAdmin.
Option 2 — Disable IPv6 Temporarily
If you disable IPv6, your system will request IPv4 instead.

Windows:

Go to Control Panel → Network and Sharing Center → Change adapter settings.
Right-click your active network → Properties.
Uncheck Internet Protocol Version 6 (TCP/IPv6).
Click OK and reconnect.
Then retry ping db.cztgpruvcuedkoqzniie.supabase.co — you should see an IPv4 address.

Option 3 — Use a VPN That Supports IPv4
Some VPNs (like ProtonVPN, NordVPN) will give you full IPv4 connectivity and bypass IPv6-only DNS issues.

Option 4 — Connect via psql with SSL over Port 443
If your network blocks IPv4 Postgres (5432) but allows HTTPS (443), Supabase supports a connection pooler on port 6543 or 443.
You can find it in:

Supabase Dashboard → Database → Connection Info → Pooler.
You can then connect in pgAdmin using:

Host: db.<project>.supabase.co
Port: 6543 or 443
SSL mode: require
